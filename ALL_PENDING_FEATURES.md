# SkillForge Complete Pending Features & Flow Audit

Based on a deep-dive scan of the entire frontend matrix and backend controllers, here are **all the incomplete features, missing workflows, and dead-end logic paths** that are currently pending in the project. 

---

## 1. Developer Take-Home Task Flow (Completely Missing)
**Status:** The most critical missing feature for the developer lifecycle.
* **Frontend:** The Dev Dashboard (`DevDashBoard.tsx`) has a green "Submit Task" button with an empty click handler `onClick={() => { /* submit task */ }}`. There is no modal to paste a repository link or upload a file. 
* **Backend:** There are API routes in `TaskController.ts` to *assign* tasks but **zero routes** designed to submit a task.
* **Reporting:** In `TaskSection.tsx` on the HR side, there is a placeholder reading "Submission Report — Coming Soon". The system cannot currently accept, store, or evaluate take-home tasks.

## 2. Code Editor Integration Crash
**Status:** Broken logic.
* **The Bug:** When HR triggers `open-code-editor` via socket, the candidate is routed to `/interview/:interviewId/code-editor`. This forces an unmount of the `InterviewRoom.tsx` component.
* **The Consequence:** Unmounting the Interview Room violently severs the WebRTC peer connection, killing video, audio, and screen sharing entirely while the candidate writes code.
* **Pending Fix:** The Code Editor needs to be rewritten as an embedded modal/side-panel *inside* `InterviewRoom.tsx` rather than as a standalone route.

## 3. Interview "End Call" & Lifecycle Transitions
**Status:** DB Schema supports it, functionality does not.
* **Missing Feature:** There is no "End Interview" button or backend endpoint for HR. The `InterviewStatus` enum has `COMPLETED`, but there is no mechanism to trigger this transition.
* **Consequence:** Interviews technically stay in the `STARTED` status forever, and HRs rely on closing the browser page to end interviews, which fails to trigger post-interview AI evaluations and report collations properly.

## 4. HR Interview Feedback Systems
**Status:** Placeholder UI only.
* **Frontend:** In `InterviewSection.tsx`, there is an explicit generic card saying "Interview Feedback — Coming in the next update. Coming Soon". 
* **Backend:** The HR has no capability to leave a standardized scorecard, star-rating, or "Hired / Rejected" decision on a candidate after an interview concludes. 

## 5. Notification Settings (Unwired)
**Status:** Dummy State
* **Frontend:** The `Notification.tsx` component provides beautiful toggles for "Interview Reminders" and "Task Submissions". 
* **Backend:** These toggles merely update a local React `useState`. There is no backend `settings` table schema for notifications, meaning preferences reset the moment the HR refreshes the page.

## 6. Real-Time Malpractice Checks
**Status:** Built but Deactivated.
* **Code Exists:** Three sophisticated hooks exist (`useMalpracticeDetection`, `useObjectDetection`, `useBrowserMalpractice`). 
* **The Reality:** All three are intentionally commented out inside `InterviewRoom.tsx`. Even if uncommented, the backend `SocketHandle.ts` only logs the event to the console or emits it to HR. It does **not** persist violations to the database, meaning there will be no malpractice history in the final candidate report.

## 7. Mobile Device Blockade
**Status:** Hard-locked
* **Frontend Logic:** The file `DesktopGaurd.tsx` forcibly blocks any device with a screen width less than `1024px`. 
* **The Note:** It displays a message saying "Mobile support coming soon". There is currently no responsive design implementation planned for candidate interviews on mobile devices. 

## 8. Notification Emails Pipeline
**Status:** Pending / Missing Triggers
* **Issue:** While "Magic Links" and "Reschedule Options" trigger emails via `nodemailer`, critical system events currently do not have email triggers despite being standard requirements.
* **Missing Targets:**
    * Emailing HR immediately when a Developer submits their technical task.
    * Emailing Developer when the 24-hour warning for task-expiry approaches.
    * Emailing Developer when their Interview has been definitively Canceled.

## 9. Code Sandbox Questions Array
**Status:** Partial implementation
* **Issue:** `QuestionController.ts` fetches up to 15 questions, where 14-15 are marked explicitly as `isLeetcode = true`. 
* **Missing Frontend Hookup:** The frontend `Codeeditor.tsx` lacks a mechanism to dynamically inject these specific LeetCode questions (and their constraints/test cases) into the view. Currently, the code editor is disconnected from the structured LeetCode question data generated by Groq.
